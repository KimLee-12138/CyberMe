## 第5张

项目背景：农业政策“看得见”，但“不一定看得懂”
数据显示：     超过 65% 的农户曾经历过“错报、漏报或不知情导致的补贴损失” 这直接佐证了宏观政策在微观落地层面的“肠梗阻”现象

## 第6张

解决方式： 主要采用基于关键词检索
传统政务信息检索系统与农业资讯门户
局限性：存在严重的“信息孤岛”与“认知壁垒”
优点：政策覆盖面广，文件来源权威（政府官网直接同步），不存在大模型的“幻觉”问题。
项目背景：农业政策“看得见”，但“不一定看得懂”

## 第7张

项目背景：当前农业政策申报场景中存在以下问题：
政策文件分布在政府网站、栏目公告、PDF附件、Word文件等多个渠道中，用户难以及时发现。
文件中大量使用行政化、专业化表达，普通农户难以准确理解“适用对象、申报条件、材料要求、时间限制”等内容。
农户需要将自身经营情况与政策条款逐项比对，容易出现漏看、错判、误报等问题。
用户不仅需要知道“能不能申报”，更需要知道“还差什么条件”“接下来该怎么补齐”。
政策来源分散
政策文本专业门槛高
人工比对效率低
申报准备缺乏路径指导

## 第8张

项目背景：主要创新点
将政策原文转化为可计算的条件树，而不是只做关键词匹配。
通过主体类型、经营面积、绿色认证、灌溉方式等信息构建用户画像。
使用确定性规则判断政策条件，输出匹配结果和失败节点，增强可解释性。
政策文本结构化
农户画像驱动匹配
白盒匹配机制
大模型主要用于政策编译、白话解释、建议生成，而不是完全替代规则判断。
大模型辅助解读
支持政策导入、审核、结构化入库、自动爬虫、任务中心和趋势分析。
后台治理闭环

## 第10张

总体需求：普通用户端 + 管理员端双角色协同
1. 普通用户（包括农户个人、家庭农场、农民合作社、农业企业等。）
主要需求：
1.创建和维护自己的经营主体画像；
2.查询与自身情况相关的农业政策；
3.自动判断是否符合申报条件；
4.查看缺口分析、风险提示和行动建议；
5.通过聊天方式进行政策追问和白话解读；
6.查看推荐政策和政策趋势信息。
2. 管理员包括政策库维护人员、系统管理人
主要需求：
1.登录后台并进行权限管理；
2.新增、编辑、删除结构化政策；
3.上传Word/PDF文件并解析政策文本；
4.使用大模型辅助编译政策条件树；
5.配置手动或自动爬虫；
6.查看后台任务运行状态；
7.管理政策风向标和洞察数据。

## 第15张

设计架构
前端：使用 Vue 3 作为前端框架，配合 Vite 构建项目，使用 Vue Router 管理页面路由，使用 Element Plus 作为 UI 组件库，使用 ECharts 做政策洞察、风向标等可视化图表展示，部分 Markdown 内容用 markdown-it 渲染。
后端：使用 Python + FastAPI 搭建后端服务，使用 Pydantic v2 做请求参数校验和响应模型约束，数据库使用 MySQL。
后台服务：使用 APScheduler 做定时任务和后台调度，例如自动爬虫、风向标生成等；接入 DeepSeek 兼容接口 用于政策规则编译、聊天增强、审核建议和报告生成；接入 百度 OCR 用于扫描版 PDF 或图片政策文件的文字识别。

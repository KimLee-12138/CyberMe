核心知识点清单

常见踩坑与注意事项

字符集陷阱: 永远使用 utf8mb4 而非 utf8，否则 emoji 和部分生僻汉字会存储失败

SQL 注入: 绝不使用字符串拼接构建 SQL，始终使用参数化查询（%s 占位符）

索引不是越多越好: 索引会降低写入速度并占用磁盘空间，只为高频查询条件建索引

事务未关闭: autocommit=False 时忘记 commit() 会导致锁等待；忘记 rollback() 会导致连接异常

Python 连接泄漏: 确保 cursor.close() 和 conn.close()，推荐使用 with 语句或 try...finally

MySQL vs 其他数据库

学习心得

对 MySQL 的认识

通过本次学习，我深刻体会到 MySQL 不仅仅是一个"存数据"的工具，而是一个功能完备的数据管理平台。MySQL 8.0 的 JSON 支持让我们可以在关系型数据库中灵活存储半结构化数据（如本项目中的政策条件树），这在实际业务中非常有用——既保留了关系型数据库的事务保证，又获得了 NoSQL 的灵活性。

理论与实践的结合

本项目的实践让我认识到，在学校课程中学到的 SQL 语法只是基础。实际项目中更需要关注的是：

数据库设计: 如何根据业务需求设计合理的表结构和索引

性能考量: 哪些查询需要优化，如何用 EXPLAIN 分析

安全防护: SQL 注入如何防护，敏感数据如何存储

技术选型: 为什么本项目在开发环境用 SQLite、生产环境用 MySQL

关键收获

参数化查询是铁律: 任何用户输入都不能直接拼入 SQL，必须通过参数传递

索引是双刃剑: 加速查询但影响写入，需要根据实际查询模式权衡

事务保证数据一致性: 在涉及多表操作时必须使用事务包裹

ORM 提升开发效率: SQLAlchemy 让 Python 开发者可以用面向对象的方式操作数据库，同时保留了原生 SQL 的灵活性

测试结果分析

发现的缺陷

通过自动化测试发现了以下值得关注的边界情况：

空用户名/短密码返回 400 而非 422（TC-REG-005, TC-REG-006）：

预期：应返回 422 Unprocessable Entity（Pydantic 验证错误标准码）

实际：返回 400 Bad Request

分析：后端在 Pydantic 验证之前进行了额外的业务校验，返回了 400 而非 422

错误管理员密码返回 400 而非 401（TC-ADM-002）：

预期：应返回 401 Unauthorized

实际：返回 400 Bad Request

分析：后端可能将密码格式校验（长度不足）优先级置于凭据校验之前

政策计数接口返回格式为 {"count": 0} 而非 0（TC-POL-002）：

实际返回：{"count": 0}（包装在 JSON 对象中）

这是合理的设计选择，测试已兼容两种格式

常见陷阱与解决方案

心得体会

1. 自动化测试是软件质量的基石

通过本次实践，我深刻体会到自动化测试的价值。手动测试 25 个 API 用例可能需要 30 分钟，而 pytest 仅需 33 秒 即可完成。更重要的是，自动化测试可以随时重复执行，在每次代码变更后快速回归验证，这是手动测试无法比拟的。

2. 测试应该覆盖正常路径和异常路径

很多初学者只写"happy path"测试（正常输入 → 正常输出），但真正的 bug 往往出现在异常路径中。本次测试特别关注了：

缺少必填字段（422）

重复注册（409）

数据库故障（503）

权限越界（403）

这些边界测试用例发现了一些实际的状态码不一致问题。

3. Selenium 测试的设计需要关注前端特性

现代 SPA 应用（Vue 3）与传统的多页面应用有很大不同：

DOM 是异步渲染的，需要显式等待。CSS class 可能是动态生成的，Element Plus 组件的实际 DOM 结构与模板不一致，因此编写 Selenium 测试时需要实际查看浏览器渲染结果来确定选择器。

4. 工具选择要匹配项目技术栈

选择 Selenium + pytest 而非 Cypress 或 Playwright，主要是因为：项目后端使用 Python、Selenium 支持 Python 绑定、一种语言覆盖前后端测试，降低维护成本

后续改进方向

数据驱动测试：使用 @pytest.mark.parametrize 将测试数据外部化到 JSON/YAML 文件

个人参与代码编写概述

在本项目中，我主要承担后端部分的代码编写工作。项目采用前后端分离架构，前端负责页面交互和展示，后端负责业务规则处理、接口服务、数据持久化、权限校验、任务调度和第三方能力接入。我围绕“农户画像—政策匹配—政策解读—政策治理—任务调度—政策洞察”的业务主线，完成了多个后端模块的接口开发与业务逻辑实现。

我的工作重点包括：编写 RESTful API 接口，设计政策匹配算法，完成条件树解析逻辑，接入 OCR 与大模型服务，支持政策文件解析、AI 规则编译、聊天 SSE 流式返回、自动爬虫、任务中心、运行观测和政策风向标等后端功能。

参与编写的主要后端功能代码

重点负责功能说明

1. 农户画像与政策匹配功能

我负责实现画像接口与匹配引擎之间的数据流转。用户在前端填写经营主体画像后，后端将画像字段保存并转换为结构化对象。政策匹配时，系统读取用户画像和结构化政策条件树，调用 DFS 匹配算法判断用户是否满足政策条件，并输出是否完全匹配、未满足条件、匹配得分和行动建议。

该部分的核心难点是保证匹配结果可解释。因此我在匹配逻辑中保留了每个条件节点的判断过程，包括规则期望值、用户实际值、是否满足和缺失原因，使前端能够展示清晰的政策体检结果。

2. 对话式政策解读功能

我负责编写聊天工作台后端接口和 SSE 流式响应逻辑。用户输入政策问题后，后端读取用户画像、历史会话和相关政策内容，构建 Prompt 后调用大模型生成回复，并通过 SSE 逐步返回给前端。该功能使用户能够以自然语言方式理解政策条款，同时结合匹配结果获得申报建议。

3. 政策文件解析与规则编译功能

在管理员导入政策文件时，我负责后端文件接收、OCR 调用、文本标准化和 AI 编译服务衔接。系统能够识别 Word、PDF、扫描件等不同文件类型，对可读取文本直接解析，对扫描类文件调用 OCR 服务提取文字，再将政策原文提交给大模型生成条件树草稿和审核任务。

4. 政策治理与审核功能

我参与实现审核任务接口，使管理员能够查看 AI 编译结果、修改条件树 JSON、保存草稿、审核通过或驳回。该流程保证政策规则不会直接进入正式政策库，而是经过人工复核后再用于用户匹配，提高了系统可靠性和可追溯性。

5. 自动爬虫、任务调度与运行观测功能

我负责编写爬虫服务、调度服务和任务观测相关后端代码。系统支持管理员配置爬虫规则，也支持定时自动抓取公开政策网站数据。每次任务运行后，系统会记录执行状态、耗时、抓取数量、失败链接和错误原因，便于管理员在任务中心查看运行情况。

6. 政策洞察与风向标功能

我参与实现政策统计、趋势分析、大模型摘要生成和智库词典服务。后端从政策库中聚合政策数量、来源分布、发布趋势和受众类型等信息，并调用大模型生成政策周报和热点摘要，为前端大屏和风向标页面提供数据。

项目历程回顾

回顾整个《农策微光》项目的开发过程，最深的感受并不是我们完成了多少个功能模块，而是从一个比较模糊的想法，一步步把它变成一个可以运行、可以演示、可以说明设计思路的软件系统。

项目刚开始时，我们其实对“农业政策匹配”这个方向并没有特别清晰的技术方案。最初大家讨论时，更多是从生活和调研中发现问题：农业政策很多，但普通用户很难看懂；政策文件里有很多条件、对象、材料要求，如果靠人工一条条比对，效率很低。于是我们最开始想做的只是一个“政策查询和解释工具”。但在进一步讨论后，我们发现仅仅做关键词搜索并不能解决核心问题，用户真正关心的是“我能不能申报”“我还差什么条件”“下一步该做什么”。也正是这个讨论过程，让我们的项目逐渐从普通的政策查询系统，变成了“农户画像 + 条件树匹配 + 大模型解读”的设计方向。

在需求分析阶段，我们也经历过功能范围不断扩大的情况。一开始大家都很有想法，希望把政策匹配、政策问答、文件导入、自动爬虫、管理员审核、政策风向标等功能都做进去。讨论的时候每个功能都觉得有意义，但真正开始拆分任务时才发现，课程项目的时间和精力是有限的。如果没有主次，很容易什么都想做，最后每个功能都做得不够完整。后来我们逐渐把主线明确为“政策从哪里来、如何变成结构化规则、如何和用户画像匹配、如何把结果解释给用户”，其他功能则围绕这条主线展开。这个过程让我第一次体会到，需求分析不是简单地把功能列出来，而是要判断哪些功能真正支撑系统核心价值。

进入设计和开发阶段后，团队分工逐渐明确。前端同学负责页面和交互，数据库同学负责表结构和数据关系，我主要负责后端接口、匹配逻辑、大模型调用、文件解析和任务调度等内容。刚开始联调时，我们经常遇到一些看似很小但很影响效率的问题，比如前端需要的字段和后端返回的字段名字不一致，数据库表中某些字段还没有确定，或者接口返回的数据结构和页面展示方式对不上。有时候一个功能在后端测试是正常的，但一接到前端页面就发现还缺少状态字段、错误提示或格式转换。经过几次这样的联调后，我们才真正意识到接口约定的重要性，也开始更重视提前沟通字段、返回格式和异常情况。

项目中比较有代表性的一次经历是政策匹配功能的实现。最开始我们只是想简单判断用户画像是否满足政策条件，但写着写着发现，如果只返回“符合”或“不符合”，其实对用户帮助不大。因为用户更需要知道自己为什么不符合，具体差在哪些条件。于是我们在匹配逻辑中加入了条件节点、必须条件、建议条件、缺失条件和行动建议等内容，让匹配结果变得更可解释。这个过程让我意识到，软件开发并不是把逻辑跑通就结束了，还要站在用户角度思考结果是否能被理解、是否有实际指导意义。

另一个印象比较深的地方是大模型和 OCR 等外部能力的接入。刚开始我们觉得调用大模型接口或 OCR 接口只是简单地发送请求、接收结果，但实际开发时发现外部服务并不总是稳定的。比如文件解析可能失败，大模型返回的内容格式可能不完全符合预期，网络异常也会导致流程中断。为了解决这些问题，我们需要增加错误提示、状态记录和人工审核流程，而不是直接把模型生成的内容当成最终结果。这个过程让我认识到，AI 能力可以提高效率，但在软件系统中仍然需要规则约束、人工复核和异常处理。

在后期测试和整理文档时，我们也遇到了一些压力。因为前期主要精力放在功能开发上，很多细节问题是在测试阶段集中暴露出来的。例如部分异常输入没有提示，任务状态显示不够清晰，接口失败时前端页面不知道如何处理，测试用例也需要不断补充。与此同时，我们还要整理需求规格说明书、概要设计、详细设计和测试报告。刚开始觉得文档只是提交材料，但写到后面发现，文档实际上是在倒逼我们重新梳理系统逻辑：这个功能为什么存在、数据从哪里来、接口如何调用、异常情况怎么处理、测试如何覆盖。通过文档整理，我们对整个系统的理解反而更加完整。

总体来看，这次项目并不是一开始就按计划顺利推进的，而是在不断讨论、修改、联调和补充中逐渐成型的。我们经历过功能想得太多、接口对不齐、测试发现问题、文档返工等情况，也在这些问题中逐渐学会了如何用软件工程的方法推进一个团队项目。相比最终系统本身，我觉得更重要的是这段开发过程让我们真正体会到：一个软件项目不是单个人写完代码就可以完成的，而是需要需求分析、架构设计、接口协作、数据库支撑、测试验证和文档记录共同配合，才能形成一个相对完整、可维护的系统。

项目中的经验与做得较好的地方

首先，本次项目的选题具有较强的现实意义。农业政策服务本身存在政策来源分散、文本专业性强、人工比对效率低等问题，我们将大模型、规则匹配、OCR、爬虫和可视化展示结合起来，尝试用软件工程方法解决一个具体场景中的实际问题，这让我感受到软件开发并不只是完成代码功能，更重要的是围绕真实需求设计可用的系统。

其次，团队在项目初期进行了较为明确的模块划分。前端、后端和数据库分工比较清晰，使每个人都能围绕自己的职责推进开发。比如前端同学主要负责 Vue 页面和交互展示，数据库同学负责核心数据表设计，我主要负责后端接口、匹配逻辑、文件解析、大模型调用和后台任务等功能。这样的分工提高了开发效率，也减少了重复劳动。

第三，项目中比较重视文档撰写。我们先后完成了需求规格说明书、概要设计说明书、详细设计说明书和测试报告等文档。通过撰写这些文档，我们对系统功能、模块接口、数据结构和测试范围有了更清晰的认识。文档不仅是课程提交材料，也在实际开发中起到了统一团队理解、明确开发边界和辅助后期测试的作用。

第四，项目后端设计中尽量体现了模块化思想。例如用户画像、政策匹配、文件导入、OCR、规则编译、审核任务、爬虫调度、任务观测和聊天解读等功能被拆分为不同服务模块，避免所有逻辑混杂在一起。这样做虽然增加了一定设计成本，但后期维护和调试更加方便，也更符合软件工程中高内聚、低耦合的思想。

第五，项目尝试引入了多种工具和技术，包括 FastAPI、Vue、MySQL、大模型接口、OCR、SSE 流式响应、ECharts 可视化等。虽然每项技术都不算特别深入，但通过项目实践，我们初步体验了一个完整软件系统从需求到实现、从接口到数据、从功能到测试的全过程。

项目中的不足与教训

虽然项目最终完成了主要功能，但回顾整个过程，也存在不少不足。

首先，前期需求边界还不够稳定。项目开始时，我们对功能设想比较多，希望同时实现政策匹配、对话问答、政策审核、文件解析、自动爬虫、风向标等模块，导致项目范围偏大。后期开发时才发现，在课程项目有限时间内完成这么多功能会带来较大压力。未来项目中应该在前期更严格地区分核心功能和扩展功能，优先保证主流程稳定，再逐步增加附加功能。

其次，接口约定在早期不够细致。前后端联调过程中，有时会出现字段命名不一致、返回格式不统一、错误状态缺少说明等问题，导致前端页面需要临时适配，后端也需要反复修改。这个问题说明接口设计不能只停留在口头沟通，而应该尽早形成接口文档，明确请求参数、返回字段、错误码和示例数据。

第三，测试工作介入得偏晚。前期我们更关注功能实现，等到后期集中测试时才发现一些异常输入、空数据、接口失败、文件解析失败、大模型不可用等边界场景处理不够完善。如果能在开发过程中同步编写测试用例，并在每个模块完成后及时测试，就可以更早发现问题，减少后期集中修改的压力。

第四，任务进度管理还可以更加规范。项目开发过程中，有些任务完成时间依赖个人进度，团队整体的进度追踪不够细致。部分模块在联调前才发现依赖接口或数据表尚未完全确定，影响了整体效率。未来应当使用更明确的任务清单或看板工具，将任务拆分到更小粒度，并定期检查完成情况。

第五，代码规范和异常处理仍有改进空间。由于项目时间紧张，部分代码更偏向“能运行”，在注释、异常分支、日志记录、函数封装和命名规范上还不够统一。尤其是涉及大模型调用、OCR、爬虫和文件解析等外部依赖的功能，更需要完善异常处理和降级方案，避免外部服务失败导致整个流程中断。

未来项目中需要注意的地方

通过这次项目，我认为以后再参与软件开发项目时，需要特别注意以下几点。

第一，需求分析阶段要控制范围。不能一开始就追求功能多，而应先明确系统最核心的业务流程。对于课程项目或时间有限的项目，应优先完成最能体现项目价值的主线功能，再考虑扩展功能。

第二，接口文档要尽早确定。前后端分离项目中，接口是团队协作的关键。后端在开发前应与前端、数据库同学共同确认字段含义、请求方式、返回结构和错误处理方式，减少后期反复联调。

第三，数据库设计要结合真实业务流程。数据表不能只根据页面临时设计，而应从业务实体和数据流转出发，提前考虑数据之间的关系、后期查询需求、状态变化和扩展字段。

第四，测试应贯穿整个开发过程。每个模块完成后都应该及时进行单元测试或功能测试，而不是等项目快结束时才集中测试。对于异常输入、权限错误、网络异常、外部服务不可用等情况，也应提前设计处理方案。

第五，团队协作要有明确的任务管理。每个人负责的功能、接口、页面和数据表都应清楚记录，定期同步进度。遇到接口变更或功能调整时，也要及时通知其他成员，避免信息不同步。

第六，文档撰写不能拖到最后。软件工程文档不是简单的总结材料，而是开发过程的一部分。需求文档、设计文档和测试文档如果能随着项目推进逐步完善，会更真实，也更有助于团队理解系统。

自己的收获与感悟

通过本次项目，我对软件开发过程有了更完整的认识。以前我对软件开发的理解更多停留在写代码层面，认为只要功能实现出来就算完成。但这次项目让我认识到，一个完整的软件工程项目不仅包括编码，还包括需求分析、系统设计、接口设计、数据库设计、文档撰写、测试验证和团队协作等多个环节。代码只是其中的一部分，真正困难的是如何让不同模块按照统一的设计协同工作。

在后端开发方面，我收获很大。通过实现用户画像接口、政策匹配接口、DFS 条件树匹配算法、聊天 SSE 接口、OCR 文件解析、政策规则编译、爬虫调度和任务观测等功能，我对后端系统的职责有了更深入的理解。后端不仅要接收请求、返回数据，还要承担业务规则处理、异常处理、数据一致性维护和外部服务整合等任务。尤其是在政策匹配功能中，我体会到算法逻辑和业务解释性同样重要，系统不仅要判断“是否匹配”，还要说明“为什么匹配”或“差在哪里”。

在文档撰写方面，我认识到软件工程文档的重要性。需求规格说明书帮助我们明确系统要做什么，概要设计说明书帮助我们理解系统整体架构，详细设计说明书帮助我们落实到具体模块和接口，测试报告帮助我们验证系统是否达到预期。通过撰写这些文档，我逐渐学会用更规范的方式描述系统，而不是只凭个人理解写代码。

在工具使用方面，我接触并实践了多种开发工具和技术，包括 FastAPI、Vue、MySQL、大模型接口、OCR、ECharts、接口调试工具和文档编辑工具等。这些工具提高了开发效率，也让我认识到现代软件开发往往不是单一技术的使用，而是多个工具、框架和服务的综合集成。

在团队协作方面，我体会到沟通和约定非常重要。前端、后端、数据库之间存在很多依赖关系，任何一个字段、接口或数据结构的变化，都可能影响其他成员的工作。因此，团队成员之间需要及时沟通，尽量提前确定规范，并在遇到问题时共同解决。相比个人独立写代码，团队项目更考验协作能力、责任意识和工程规范。

总体来说，这次项目让我从软件工程学生的角度，较完整地经历了一次项目开发过程。虽然项目仍有不足，但它让我真正理解了需求、设计、编码、测试和文档之间的关系，也让我意识到软件工程强调的不只是“写出程序”，更是“用规范的方法开发可维护、可协作、可扩展的软件系统”。这对我今后继续学习软件开发和参与实际项目都有很大帮助。

# 第4章：结构化分析

> cover：小孩不哭站起来lu

## 一、结构化分析基本思想

**（1）核心特点：**

- **采用自顶向下、逐层分解的方法求解复杂问题。**
- 结构化分析方法是一种传统系统建模技术。
- 它把系统建模成信息内容和数据流的模型、数据和控制行为模型、数据结构模型。

**（2）需求分析任务：**

- 需求分析任务是准确地指出：**“软件目标产品必须做什么？”**

## 二、结构化分析模型

**（1）分类：**

- **面向系统功能的模型：**数据流模型、状态转换模型、实体关系模型。
- **面向数据的模型：**实体关系模型。

**（2）结构化分析模型组成：**

- **数据字典（DD，Data Dictionary）：**分析模型核心，描述数据流图中所有名字的集合及其定义。
- **数据流图（DFD，Data Flow Diagram）：**描述数据在系统中的流动与加工。
- **实体-关系图（E-R 图）：**描述数据对象、属性及对象之间的关系。
- **状态转换图（STD）：**描述系统状态和状态转换。

## 三、面向数据流的建模方法

**（1）建模工具：**

- **数据流图 DFD。**
- **数据字典 DD。**

**（2）构建原则：**

- **自顶向下、逐层分解。**

（3）优点：

- 使问题复杂度逐层降低。
- 有利于系统逻辑功能的清晰描述。
- 方便功能分解与实现。

## 四、数据流图（DFD）

**（1）定义：**

- 数据流图是一种图形化技术，用于描述信息流和数据在系统中的移动。
- 它反映系统的功能域。

## 五、数据流图构建步骤

**（1）构建顶层数据流图：**

- 明确系统的主要输入和输出。
- 从整体角度表示系统与外部实体之间的数据交换。

**（2）构建 0 层数据流图：**

- 将顶层图的核心加工分解为若干主要加工。
- 建立主要数据流、数据存储和外部实体之间的关系。

**（3）逐层细化：**

- 对 0 层图中的每个加工继续细化。
- 分别构建 1 层图、2 层图，直到能够清楚描述底层处理。

**（4）总体原则：**

- 先全局，后局部。

- 先整体，后细节。

- 由抽象到具体，逐层细化。

## 六、数据流图注意事项

**（1）命名：**

- 数据流图中的元素必须有名字，且命名合理。
- 加工名通常应反映处理动作。

**（2）保持平衡：**

- 父图与子图之间的数据输入和输出应保持一致。
- 子图不能凭空增加或丢失父图中没有的数据流。

**（3）加工约束：**

- 每个加工至少有一个输入数据流和一个输出数据流。
- 不能只有输入没有输出，也不能只有输出没有输入。

**（4）编号一致：**

- 子图编号应继承父图加工编号。
- 编号用于表示层次关系。

## 七、数据字典

**（1）定义：**

- 数据字典是分析模型中出现的所有名字的集合，并包含有关命名实体的描述。
- 数据字典是**分析模型的核心**。

**（2）作用：**

- 定义系统中的数据含义。
- 保证开发人员对数据理解一致。
- 与数据流图配合使用。

**（3）四类条目：**

- **数据流条目。**
- **数据存储条目。**
- **数据项条目。**
- **加工条目。**

## 八、面向数据的建模方法：E-R 图

**（1）基本作用：**

- 描述数据对象、对象属性以及对象之间的关系。

**（2）关键概念：**

- **实体。**
- **属性。**
- **联系。**

**（3）关系类型：**

- 一对一。
- 一对多。
- 多对多。

**（4）适用：**

- 更侧重系统的数据结构和数据对象之间的关系。

## 九、面向状态的建模方法：状态转换图

**（1）定义：**

- 状态转换图描述系统对内部或外部事件响应的行为。

**（2）作用：**

- 描述系统状态、事件以及事件引发的状态间转换。

**（3）组成：**

- 状态。
- 事件。
- 转换。
- 初态。
- 终态。

**（4）适用：**

- 适合描述实时系统或状态变化明显的系统。

**（5）事件表达式：**

- **事件说明[守卫条件]/动作表达式。**

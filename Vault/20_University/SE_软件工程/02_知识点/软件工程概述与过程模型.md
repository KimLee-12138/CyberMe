# 第1章：软件工程概述

> cover：小孩不哭站起来lu

## 一、什么是软件

**（1）软件定义：**

- **软件是计算机系统中与硬件相对应的另一部分，是一系列程序、数据以及相关文档的集合。**
- 软件不是只指程序，课件与复习资料都强调：软件 = **程序 + 数据 + 文档**。

**（2）软件三要素：**

- **程序：**数据和指令的集合。
- **数据：**数据结构。
- **文档：**有关的资料。

**（3）软件分类：**

- **系统软件：**如操作系统、设备驱动程序。
- **支撑软件：**如工具软件、中间件。
- **应用软件：**特定应用领域的软件。

## 二、软件的特性

**（1）复杂性（Complexity）：**

- 大型软件可能包含百万行源代码、上万个文件。
- 模块之间存在大量显性和隐性的依赖关系。

**（2）不可见性（Invisibility）：**

- 软件的具体执行过程不可直接观察。
- 软件出错时，很难完整重现程序出现问题的过程。

**（3）服从性 / 一致性（Conformity）：**

- 软件需要服从硬件。
- 软件需要服从用户需求、行业系统要求。

**（4）易变性（Changeability）：**

- 软件经常要升级，以支持新功能或适应新硬件。
- 软件比硬件容易修改，但要正确修改并不容易。

## 三、软件危机

**（1）定义：**

- **软件危机是软件开发和维护过程中所遇到的严重问题。**

**（2）解决途径：**

- **重视需求分析，与客户沟通，确切表达需求。**
- **重视设计和实现过程的资料。**
- **进行充分的测试工作。**
- **做好维护和管理工作。**
- **用统一的、公认的方法论和规范指导软件开发。**

## 四、什么是软件工程

**（1）软件工程三要素：**

- **方法：**提供技术手段。
- **工具：**提供支撑环境。
- **过程：**规定任务框架和步骤。

**（2）软件工程管理三要素：**

- **成本。**
- **质量。**
- **工期。**

## 五、软件开发的本质

- 软件开发的本质是：**实现问题域中的概念和处理逻辑到运行平台的概念和处理逻辑的映射。**
- 控制软件复杂性的基本方法是：**抽象（建模）与分层。**

## 七、软件工程基本原则

- **分阶段的软件开发**
- **坚持进行阶段评审**
- **实行严格的产品控制**
- **采用先进的程序设计技术**
- **明确责任**
- **开发小组成员应少而精。**
- **不断改进开发过程。**

## 八、软件工程开发方法学

**（1）结构化开发方法学：**

- 特征：**自顶向下、逐级细化、过程式。**
- 过程式强调：输入到输出的变换。
- 技术：结构化分析、结构化设计、结构化编程、结构化测试、结构化维护。
- 适用：小型系统，需求较稳定，系统要么注重数据，要么注重行为。

**（2）面向对象开发方法学：**

- 特征：**将对象视作封装了数据及操作的统一软件组件。**
- 技术：面向对象分析、面向对象设计、面向对象编程、面向对象测试、面向对象维护。
- 优势主要体现在维护阶段：对象概念符合业务领域实际，封装与复用使维护和扩展更方便。

**（3）重型软件工程：**

- 典型代表：**瀑布模型。**
- 特点：正规、严谨，文档齐全，周期长，规范化和程式化，强调推迟实现。

**（4）轻型软件工程：**

- 典型代表：**敏捷开发。**
- 特点：非正式交流，开发效率高，文档不多，重视代码、测试和响应需求变化。


# 第2章：软件过程及其模型

## 一、软件生存周期

**（1）定义：**

- 软件也有一个从生到死的过程，这个过程一般称为**软件的软件生存周期或生命周期**。

**（2）三个时期：**

- **软件定义时期。**
- **软件开发时期。**
- **软件运行时期。**

**（3）具体阶段：**

- **可行性分析。**
- **项目计划。**
- **需求分析。**
- **软件设计。**
- **编码与测试。**
- **运行与维护。**

## 二、软件过程

**（1）定义：**

- **软件过程是软件生存周期中的一系列相关过程所涉及的活动。**

**（2）层次关系：**

- **过程是活动的集合。**
- **活动是任务的集合。**
- **任务是将输入转换为输出的操作。**

**（3）软件过程框架活动：**

- **沟通。**
- **计划。**
- **建模。**
- **构建：**编码和测试活动。
- **部署：**将软件部署到运行环境并交付给用户。

**（4）软件工程活动：**

- 沟通、计划、建模、实现、部署、维护、管理、过程改进活动。

## 三、软件过程模型

**（1）IEEE 12207.0-1996 定义：**

- 软件过程模型是一个包括软件产品开发、运行和维护中有关过程、活动和任务的框架，覆盖从系统需求定义到系统使用终止。

**（2）基本含义：**

- 软件过程模型是软件过程的抽象表示，也称为**软件开发模型**或**软件生存周期模型**。
- 一般用直观**图形**标识软件开发过程。
- 过程模型需要根据**软件类型、规模、开发方法、开发环境**等因素确定。

**（3）三者区别：**

- **软件生存周期：**软件从生到死的整个过程。
- **软件生存周期过程 / 软件过程：**整个过程中涉及的一系列活动。
- **软件生存周期模型 / 软件过程模型：**对软件过程的抽象表示。

## 四、传统软件过程模型

**（1）传统模型包括：**

- **瀑布模型。**
- **原型模型。**
- **增量模型。**
- **螺旋模型。**

**（2）面向对象软件过程模型包括：**

- **构件集成模型。**
- **统一过程模型。**

## 五、瀑布模型

**（1）基本含义：**

- 它把软件生命周期划分为：**软件计划、需求分析与定义、设计、实现、测试、运行与维护**等阶段。

- 各阶段自上而下、相互衔接、固定次序，如同瀑布流水逐级下落。

  ![image-20260607135416923](C:\Users\22067\AppData\Roaming\Typora\typora-user-images\image-20260607135416923.png)

**（2）特点：**

- **顺序性和依赖性。**
- **推迟实现。**
- **质量保证观点。**
- **线性模型。**
- **强调文档作用。**

**（4）局限性：**

- 软件生存周期末期才能看到开发成果，风险大，错误代价大。
- 不能很好地适应需求变化。

**（5）适用范围：**

- 产品定义非常稳定，产品正确性非常重要，需要每一步验证。
- 产品模块接口、输入输出能够用形式化方法定义和验证。
- 技术成熟，团队成员能够熟练掌握相关技术。
- 负责人不同时，文档能作为沟通依据。

## 六、原型模型

**（1）基本含义：**

- 原型模型利用原型法技术尽可能快地构造一个实际系统的简化模型。

**（2）特点：**

- **能够快速实现系统初步模型。**

- **便于开发人员和用户交流，明确需求。**

- **通过逐步求精使原型逐步完善。**

- **符合人类认知过程和思维活动。**

- **适用于不能预先确切定义需求的软件项目。**

  ![image-20260607135538081](C:\Users\22067\AppData\Roaming\Typora\typora-user-images\image-20260607135538081.png)

## 七、增量模型

**（1）基本含义：**

- 增量模型也称为**渐增模型**，是在项目开发过程中以一系列增量方式开发系统。
- 软件被作为一系列的**增量构件**来设计、实现、集成和测试。

**（2）两种增量形式：**

- **增量构造模型。**
- **演化提交模型**

**（3）增量构造模型：**

- **某些阶段按瀑布模型整体方式开发**。

- **某些阶段按增量方式开发**。

- 优点：开发中用户可以及早看到部分软件功能，及时发现问题。

  ![image-20260607135615880](C:\Users\22067\AppData\Roaming\Typora\typora-user-images\image-20260607135615880.png)

**（4）演化提交模型：**

- 在瀑布模型基础上，所有阶段进行增量开发。

- 先对某部分功能进行需求分析，再设计、编码、测试并提交，直到全部功能开发完毕。

  ![image-20260607135629320](C:\Users\22067\AppData\Roaming\Typora\typora-user-images\image-20260607135629320.png)

## 八、螺旋模型

**（1）基本含义：**

- 它结合了**瀑布模型**和**快速原型模型**。

- 它强调其他模型忽略的**风险分析**。

  ![image-20260607135704623](C:\Users\22067\AppData\Roaming\Typora\typora-user-images\image-20260607135704623.png)

**（2）风险分析内容：**

- **风险识别。**
- **风险分析。**
- **风险控制。**

**（3）四个象限活动：**

- **制定计划：**确定软件目标、选定实施方案、确定项目限制条件。
- **风险分析：**分析所选方案，考虑如何识别并消除风险。
- **实施工程：**实施软件开发和验证。
- **客户评估：**评价开发工作，提出修正建议，制定下一步计划。

**（4）特点：**

- 每一个周期包括：需求定义、风险分析、工程实现和评审。
- 每迭代一次，软件开发前进一个层次。
- 螺旋模型是**风险驱动模型**。
- 特别适合**大型复杂系统**。

## 九、构件集成模型

**（1）基本含义：**

- 构件集成模型利用**模块化方法将整个系统模块化**，并复用构件库中的软件构件。
- 通过组合手段提高应用软件系统过程的效率和质量。

**（2）构件复用的方式：**

- **整个系统模块化。**
- **复用构件库中的软件构件。**

**（3）五个阶段：**

- **软件的需求分析和定义。**

- **体系结构设计。**

- **构件库建立。**

- **应用软件构建。**

- **测试和发布。**

  ![image-20260607135835322](C:\Users\22067\AppData\Roaming\Typora\typora-user-images\image-20260607135835322.png)

**（4）特点：**

- 开发过程就是构件组装过程。
- 维护过程就是构件升级、替换和扩充过程。
- 可以提高软件重用效率。

## 十、统一过程模型

**（1）基本含义：**

- 用例驱动的，以架构为中心，迭代和增量的过程

- 它是一个**二维的软件开发模型**。

**（2）四个阶段：**

- **初始阶段：**分析系统大概构成、系统与外部系统边界、成本预算、风险识别，达到生命周期目标里程碑。

- **细化阶段：**分析问题领域，建立体系结构，编制项目计划，建立支持环境，达到生命周期体系结构里程碑。

- **构造阶段：**开发所有功能集，将功能集成为经过测试验证的产品，达到初始功能里程碑。

- **交付阶段：**基于用户反馈修改和调整系统，达到产品发布里程碑。

  ![image-20260607135930919](C:\Users\22067\AppData\Roaming\Typora\typora-user-images\image-20260607135930919.png)

**（3）六个核心工作流：**

- **业务建模。**
- **需求。**
- **分析和设计。**
- **实现。**
- **测试。**
- **部署。**

**（4）总体特征：**

- **用例驱动。**
- **以架构为中心。**
- **迭代和增量。**


# 第3章：软件需求分析

## 一、需求分析定义与目的

**（1）定义：**

- 需求分析是指在开发一个新的或升级一个已有的软件系统时，描写新系统的**目的、范围、定义和功能**时所要做的所有工作。

## 二、软件需求分类

**（1）业务需求（Business requirement）：**

- 反映组织机构或客户对系统、产品高层次的目标要求。
- 从组织或客户角度看，是业务建设方的目标。

**（2）用户需求（User requirement）：**

- 用户的目标，或用户要求系统必须能完成的任务。
- 从使用产品的用户角度看。

**（3）功能需求（Functional requirement）：**

- 规定开发人员必须在产品中实现的软件功能。
- 用户利用这些功能完成任务，满足业务需求。

**（4）非功能需求（Non-functional requirements）：**

- 主要与系统总体特征相关。
- 是一些限制性要求，是对实际使用环境所做的要求。
- 包括：**性能、外部接口、设计约束、质量属性**等。

## 三、功能需求

**（1）描述方式：**

- **文字描述。**
- **图表表示。**

**（2）功能需求要点：**

- 可以以不同详细程度反复编号和细化。
- 描述应完整、一致、准确。

**（3）三项质量要求：**

- **完整性：**所有需要的服务都应给出描述。
- **一致性：**需求描述不能前后矛盾。
- **准确性：**不能出现模糊和二义性的地方。

## 四、非功能需求

**（1）主要类型：**

- **性能。**
- **外部接口。**
- **设计约束。**
- **质量属性。**

**（2）性能：**

- 系统完成规定功能时应表现出的及时性。
- 可通过响应时间、吞吐量、并发数、资源利用率等指标描述。

**（3）外部接口：**

- 规定系统必须与之交互的软件、硬件或数据库元素。
- 包括：用户接口、硬件接口、软件接口、通信接口。

**（4）设计约束：**

- 限制系统或系统构件设计方案的要求。
- 例如系统必须用某种语言开发、指定页面效果、指定字体等。

**（5）质量属性：**

- 规定软件产品必须具有的性质是否达到质量方面期望。
- 包括：可靠性、可用性、安全性、移植性等。

**（6）可靠性指标：**

- 可用 **平均故障间隔时间（MTBF）** 和 **平均修复时间（MTTR）** 评价。

**（7）可用性：**

- 计算公式：**Availability = MTBF / (MTBF + MTTR)**。
- 常用于系统可用性评价。

**（8）安全性与移植性：**

- **安全性：**系统在受到攻击时仍能维持运行，防止未授权访问。
- **移植性：**系统从一个环境移到另一个环境的容易程度。

## 五、需求分析过程

**（1）过程包括：**

- **沟通。**
- **导出需求。**
- **与客户和用户协商。**
- **可行性研究。**
- **精化需求。**
- **编写需求规格说明。**
- **验证需求。**
- **管理需求。**

## 七、需求获取技术

**（1）主要技术：**

- **会谈技术。**
- **调查技术。**
- **场景分析技术。**
- **调研相关项目。**
- **快速原型法。**

**（2）会谈技术：**

- **非正式会谈：**提出一些可自由回答的问题。
- **正式会谈：**提出一些事先准备好的议题。
- 常用多视点分析。

**（3）调查技术：**

- 确定调查内容：非正式会谈、制定调查表、组织调查。
- 可行可信分析：检查回答指标是否合理，指标之间是否存在关联，结果是否可信。

**（4）问卷调查方式：**

- 二项选择题。
- 多项选择题。
- 顺位选择题。
- 开放式问题。

**（5）场景分析技术：**

- 场景是用户与软件系统交互的过程。
- 场景描述用户输入、系统响应、异常处理、完成后的系统状态等。
- 情景开发是迭代过程，细节逐渐增加，直到产生交互的完整描述。

**（6）相关项目调研：**

- 项目调研：商业、开源。
- 文献调研：综述论文、普通论文、技术白皮书。
- 与相关项目比较：启发、凝练问题、获取需求、针对不足提出解决方案或创新。


# 第4章：结构化分析

## 一、结构化分析基本思想

**（1）核心特点：**

- **采用自顶向下、逐层分解的方法求解复杂问题。**
- 结构化分析方法是一种传统系统建模技术。
- 它把系统建模成信息内容和数据流的模型、数据和控制行为模型、数据结构模型。

**（2）需求分析任务：**

- 需求分析任务是准确地指出：**“软件目标产品必须做什么？”**

## 二、结构化分析模型

**（1）分类：**

- **面向系统功能的模型：**数据流模型、状态转换模型、实体关系模型。
- **面向数据的模型：**实体关系模型。

**（2）结构化分析模型组成：**

- **数据字典（DD，Data Dictionary）：**分析模型核心，描述数据流图中所有名字的集合及其定义。
- **数据流图（DFD，Data Flow Diagram）：**描述数据在系统中的流动与加工。
- **实体-关系图（E-R 图）：**描述数据对象、属性及对象之间的关系。
- **状态转换图（STD）：**描述系统状态和状态转换。

## 三、面向数据流的建模方法

**（1）建模工具：**

- **数据流图 DFD。**
- **数据字典 DD。**

**（2）构建原则：**

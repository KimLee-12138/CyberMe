# SignalTap II 测试案例集

本文档给出 `DE1_SOC_golden_top` 工程中 ALU 的 SignalTap II 测试案例。所有案例都基于当前工程已有的 `stp1.stp` 和 `output_files/DE1_SOC_golden_top.sof`。

## 1. 测试前统一准备

### 1.1 必要文件

| 文件 | 用途 |
| --- | --- |
| `DE1_SOC_golden_top.qpf` | Quartus 工程文件 |
| `output_files/DE1_SOC_golden_top.sof` | 下载到 FPGA 的配置文件 |
| `stp1.stp` | SignalTap II 观测配置 |

### 1.2 SignalTap II 基本设置

打开 `Tools -> SignalTap II Logic Analyzer` 后，确认：

| 项目 | 正确值 |
| --- | --- |
| JTAG Chain | 能识别 DE1-SoC / USB-Blaster |
| `.sof` 文件 | `output_files/DE1_SOC_golden_top.sof` |
| 采样时钟 | `CLOCK_50` |
| 采样边沿 | 上升沿 |
| 采样深度 | 256 |

### 1.3 推荐观察信号

每个案例建议至少观察：

```text
KEY[0]
SW[4]
alu:alu_inst|a[31..0]
alu:alu_inst|b[31..0]
alu:alu_inst|aluc[3..0]
alu:alu_inst|r[31..0]
alu:alu_inst|z
alu:alu_inst|d_as[31..0]
alu:alu_inst|d_sh[31..0]
alu:alu_inst|d_and[31..0]
alu:alu_inst|d_or[31..0]
alu:alu_inst|d_xor[31..0]
alu:alu_inst|d_xor_lui[31..0]
```

显示进制建议：

| 信号 | 显示方式 |
| --- | --- |
| `a`、`b`、`r`、`d_*` | Hexadecimal |
| `aluc`、`KEY[0]`、`SW[4]`、`z` | Binary |

## 2. 通用复位步骤

后续多数案例都使用固定输入：

```text
a = 0x0000000A
b = 0x0000000B
```

获得固定输入的方法：

1. 设置 `SW[4]=0`。
2. 按下 `KEY[0]`。
3. 保持约 1 秒。
4. 松开 `KEY[0]`。
5. 点击 SignalTap II 的 `Run Analysis`。
6. 确认：

```text
alu:alu_inst|a[31..0] = 0x0000000A
alu:alu_inst|b[31..0] = 0x0000000B
```

如果不是这两个值，重新执行以上复位步骤。

## 3. 固定输入下的 16 个 ALU 功能案例

复位后固定：

```text
a = 0x0000000A
b = 0x0000000B
a[4:0] = 10
SW[4] = 0
```

`SW[3:0]` 对应 `aluc[3:0]`：

```text
SW[3] -> aluc[3]
SW[2] -> aluc[2]
SW[1] -> aluc[1]
SW[0] -> aluc[0]
```

每个案例的通用操作：

1. 设置 `SW[4]=0`。
2. 按 `KEY[0]` 复位，确认 `a=0xA`、`b=0xB`。
3. 按案例设置 `SW[3:0]`。
4. 点击 `Run Analysis`。
5. 观察 `aluc`、`r`、`z`。
6. 对照预期值。

## 5. 中间信号观察案例

### 案例 17：观察加减法通路 `d_as`

目的：验证 `aluc[2]` 如何控制 `addsub32` 做加法或减法。

操作：

1. 设置 `SW[4]=0`。
2. 复位，确认 `a=0xA`，`b=0xB`。
3. 设置 `SW[3:0]=0000`。
4. 采样，观察：

```text
d_as = 0x00000015
r    = 0x00000015
```

5. 设置 `SW[3:0]=0100`。
6. 再采样，观察：

```text
d_as = 0xFFFFFFFF
r    = 0xFFFFFFFF
```

预期结论：

```text
aluc[2]=0 时，d_as = a + b
aluc[2]=1 时，d_as = a - b
```

### 案例 18：观察逻辑通路 AND/OR

目的：验证 `d_and_or` 如何在 AND 和 OR 之间切换。

操作：

1. 固定 `a=0xA`，`b=0xB`。
2. 设置 `SW[3:0]=0001`，采样。
3. 观察：

```text
d_and = 0x0000000A
d_or  = 0x0000000B
r     = 0x0000000A
```

4. 设置 `SW[3:0]=0101`，采样。
5. 观察：

```text
d_and = 0x0000000A
d_or  = 0x0000000B
r     = 0x0000000B
```

预期结论：

```text
aluc[2]=0 时，d_and_or 选择 d_and
aluc[2]=1 时，d_and_or 选择 d_or
```

### 案例 19：观察 XOR/LUI 通路

目的：验证 `d_xor_lui` 如何在 XOR 和 LUI 之间切换。

操作：

1. 固定 `a=0xA`，`b=0xB`。
2. 设置 `SW[3:0]=0010`，采样。
3. 观察：

```text
d_xor     = 0x00000001
d_xor_lui = 0x00000001
r         = 0x00000001
```

4. 设置 `SW[3:0]=0110`，采样。
5. 观察：

```text
d_xor_lui = 0x000B0000
r         = 0x000B0000
```

预期结论：

```text
aluc[2]=0 时，d_xor_lui 选择 d_xor
aluc[2]=1 时，d_xor_lui 选择 LUI 结果
```

### 案例 20：观察移位通路 `d_sh`

目的：验证 `shift` 模块的左移、逻辑右移、算术右移。

操作：

1. 固定 `a=0xA`，`b=0xB`。
2. 设置 `SW[3:0]=0011`，采样。
3. 观察：

```text
d_sh = 0x00002C00
r    = 0x00002C00
```

4. 设置 `SW[3:0]=0111`，采样。
5. 观察：

```text
d_sh = 0x00000000
r    = 0x00000000
z    = 1
```

6. 设置 `SW[3:0]=1111`，采样。
7. 观察：

```text
d_sh = 0x00000000
r    = 0x00000000
z    = 1
```

预期结论：

```text
0011 是左移，0111 是逻辑右移，1111 是算术右移。
由于当前 b=0x0000000B 是正数，所以逻辑右移和算术右移结果相同。
```

### 案例 21：观察最终多路选择器

目的：证明 ALU 内部先并行算出多个候选结果，再由 `aluc[1:0]` 选择最终 `r`。

操作：

1. 固定 `a=0xA`，`b=0xB`。
2. 依次设置以下控制码并采样：

| `aluc` | 选择通路 | 预期 `r` |
| --- | --- | --- |
| `0000` | `d_as` | `0x00000015` |
| `0001` | `d_and_or` | `0x0000000A` |
| `0010` | `d_xor_lui` | `0x00000001` |
| `0011` | `d_sh` | `0x00002C00` |

3. 每次同时观察：

```text
d_as
d_and_or
d_xor_lui
d_sh
r
aluc
```

预期结论：

```text
aluc[1:0]=00 时 r=d_as
aluc[1:0]=01 时 r=d_and_or
aluc[1:0]=10 时 r=d_xor_lui
aluc[1:0]=11 时 r=d_sh
```

## 6. 顶层输入行为测试案例

### 案例 22：测试 `KEY[0]` 复位

目的：确认按下 `KEY[0]` 后，`a`、`b` 回到固定初值。

操作：

1. 设置 `SW[4]=1`，让 `a`、`b` 自动变化一段时间。
2. 设置 `SW[4]=0`，停止变化。
3. 采样，记录当前 `a`、`b`，此时通常不是 `0xA`、`0xB`。
4. 按下 `KEY[0]`。
5. 松开 `KEY[0]`。
6. 再采样。

预期结果：

```text
a = 0x0000000A
b = 0x0000000B
```

判定标准：

```text
按下并松开 KEY[0] 后，a/b 能恢复到复位初值。
```

### 案例 23：测试 `SW[4]=0` 保持数据

目的：确认 `SW[4]=0` 时，`a` 和 `b` 不会自动递增。

操作：

1. 设置 `SW[4]=0`。
2. 按下并松开 `KEY[0]`。
3. 连续点击多次 `Run Analysis`。
4. 观察每次采样中的 `a` 和 `b`。

预期结果：

```text
a 始终为 0x0000000A
b 始终为 0x0000000B
```

判定标准：

```text
多次采样后 a/b 不变化。
```

### 案例 24：测试 `SW[4]=1` 自动递增

目的：确认 `SW[4]=1` 时，`pattern_a` 每个时钟加 1，`pattern_b` 每个时钟加 3。

操作：

1. 设置 `SW[4]=0`。
2. 按下并松开 `KEY[0]`。
3. 设置 `SW[3:0]=0000`，选择 ADD。
4. 把 `SW[4]` 拨到 1。
5. 点击 `Run Analysis`。
6. 观察连续采样点中的 `a`、`b`、`r`。

预期变化规律：

```text
a: 每个采样点加 1
b: 每个采样点加 3
r: 当前为 ADD，所以 r = a + b
```

示例趋势：

| 采样点 | `a` | `b` | ADD 结果 `r` |
| --- | --- | --- | --- |
| n | `A0` | `B0` | `A0+B0` |
| n+1 | `A0+1` | `B0+3` | `A0+B0+4` |
| n+2 | `A0+2` | `B0+6` | `A0+B0+8` |

注意：实际第一个采样点不一定是 `0xA`、`0xB`，因为打开 `SW[4]` 到采样开始之间已经过了很多 50 MHz 时钟周期。

## 8. 加减法器内部测试案例

### 案例 27：观察减法时 `b` 被取反

目的：验证 `addsub32` 通过 `a + ~b + 1` 实现减法。

操作：

1. 固定 `a=0xA`，`b=0xB`。
2. 设置 `SW[3:0]=0100`。
3. 采样。
4. 观察：

```text
alu:alu_inst|addsub32:as32|a[31..0]
alu:alu_inst|addsub32:as32|b[31..0]
alu:alu_inst|addsub32:as32|s[31..0]
```

预期：

```text
addsub32:as32|a = 0x0000000A
addsub32:as32|b = 0xFFFFFFF4
addsub32:as32|s = 0xFFFFFFFF
```

解释：

```text
0x0000000B 取反 = 0xFFFFFFF4
a + ~b + 1 = 0x0000000A + 0xFFFFFFF4 + 1 = 0xFFFFFFFF
```

## 9. 触发观察测试案例

### 案例 28：触发捕获复位释放

目的：捕获 `KEY[0]` 从按下到松开的变化。

操作：

1. 在 SignalTap II 中把 `KEY[0]` 加入触发条件。
2. 设置触发条件为 `KEY[0]=1`，或设置为上升沿触发。
3. 点击 `Run Analysis`。
4. 按下 `KEY[0]`。
5. 松开 `KEY[0]`。
6. 等待采样完成。

预期：

```text
KEY[0] 由 0 变为 1 后，a 和 b 回到或保持复位值：
a = 0x0000000A
b = 0x0000000B
```

### 案例 29：触发捕获 `SW[4]` 打开自动递增

目的：捕获 `SW[4]` 从 0 到 1 后，`a` 和 `b` 的连续变化。

操作：

1. 设置 `SW[4]=0`。
2. 复位。
3. 在 SignalTap II 中把 `SW[4]` 加入触发条件。
4. 设置触发条件为 `SW[4]=1`，或设置为上升沿触发。
5. 点击 `Run Analysis`。
6. 把 `SW[4]` 拨到 1。
7. 等待采样完成。

预期：

```text
SW[4]=1 后：
a 连续递增
b 连续递增且步长为 3
```

## 10. 故障排查测试案例

### 案例 30：SignalTap 节点无效检查

现象：

```text
SignalTap 中节点变红、invalid、node not found
```

检查步骤：

1. 确认当前打开工程是 `DE1_SOC_golden_top.qpf`。
2. 确认打开的是 `stp1.stp`。
3. 确认 `.sof` 指向 `output_files/DE1_SOC_golden_top.sof`。
4. 重新下载 `.sof`。
5. 重新打开 SignalTap II。
6. 如果仍无效，用 Node Finder 重新查找：

```text
alu:alu_inst|a[31..0]
alu:alu_inst|b[31..0]
alu:alu_inst|aluc[3..0]
alu:alu_inst|r[31..0]
alu:alu_inst|z
```

判定标准：

```text
节点恢复有效，Run Analysis 后能看到采样值。
```

### 案例 31：结果与理论表不一致排查

现象：

```text
r 和预期结果不一致
```

按顺序检查：

1. `SW[4]` 是否为 0。
2. `a` 是否为 `0x0000000A`。
3. `b` 是否为 `0x0000000B`。
4. `aluc[3..0]` 是否等于当前案例要求。
5. `r` 是否用 Hexadecimal 显示。
6. `.sof` 是否为 `output_files/DE1_SOC_golden_top.sof`。
7. 是否误把 `SW[0]` 和 `SW[3]` 顺序看反。

最常见错误：

```text
想设置 aluc=0100，但实际开关拨成了 0010。
```

解决：

```text
直接看 SignalTap 中的 aluc[3..0]，不要只凭肉眼猜开关。
```

## 硬连线路设计和微程序设计

- 控制器设计有两种方法：硬连线路设计和微程序设计。


硬连线路设计：

- 用组合逻辑电路直接根据指令 op、func 等字段产生控制信号。
- 优点是**速度快，适合简单、规整的指令系统**，如 MIPS；
- 缺点是**结构复杂，修改维护困难，灵活性差**。

微程序设计：

- 把每条机器指令的执行过程编写成一个微程序。
- 每个微程序由若干条微指令组成，每条微指令包含若干微命令。
- 微命令本质上就是控制信号。
- 执行时，控制器依次从控制存储器中取出微指令，并产生相应控制信号。

控制存储器 Control Storage，简称 CS，用于存放微程序。

以 lw rt, rs, imm16 为例，其执行可分为：
1. 取指令：IR ← M[PC]
2. 计算地址：Addr ← R[rs] + SignExt(imm16)
3. 读内存并写回：R[rt] ← M[Addr]
4. 更新 PC：PC ← PC + 4

微程序设计的特点是**灵活、易修改，适合复杂指令系统**；
但由于**需要逐条取出微指令执行，因此速度较慢**。

## 微程序控制器的基本结构

![img](C:\Users\22067\Desktop\计组复习\assets\cover470_20260511075957.jpg)

微程序控制器通过执行微程序来产生控制信号。

输入：

- 机器指令、条件码。

输出：

- 控制信号，即微命令。

CS：

- 控制存储器，用于存放微程序。

μPC：

- 微程序计数器，保存下一条要执行的微指令在 CS 中的地址。

μIR：

- 微指令寄存器，保存当前正在执行的微指令。

起始地址发生器：

- 根据当前机器指令产生该指令对应微程序的第一条微指令地址。

执行过程：

1. 根据 μPC 从 CS 中取出微指令送入 μIR；
2. μIR 经译码产生控制信号；
3. 每个时钟执行一条微指令；
4. 顺序执行时 μPC←μPC+1；
5. 转移执行时根据转移控制字段和条件码修改 μPC。

固化在 ROM 中的微程序称为固件 Firmware。

### 一、不译法：直接控制法

#### 1. 基本思想

不译法最简单粗暴：

> **一位对应一个微命令，也就是一个控制信号。**

比如 CPU 里有这些控制信号：

```
PCout
MARin
Read
MDRin
IRin
RegWrite
ALUadd
ALUsub
MemWrite
```

那么微指令中就直接给每个控制信号分配一位：

| 位   | 控制信号 |
| ---- | -------- |
| bit0 | PCout    |
| bit1 | MARin    |
| bit2 | Read     |
| bit3 | MDRin    |
| bit4 | IRin     |
| bit5 | RegWrite |
| bit6 | ALUadd   |
| bit7 | ALUsub   |
| bit8 | MemWrite |

如果某一位是 `1`，就表示这个控制信号有效；如果是 `0`，就表示无效。

------

#### 2. 举例：实现 `PC → MAR`

如果这一拍要执行微操作：

```
PC → MAR
```

需要打开两个控制信号：

```
PCout = 1
MARin = 1
```

其他信号不用。

那么微指令对应的控制位大概就是：

```
PCout MARin Read MDRin IRin RegWrite ALUadd ALUsub MemWrite
  1     1    0    0    0      0      0      0       0
```

也就是：

```
110000000
```

这个微指令不需要再经过复杂译码，控制信号可以直接输出。

------

#### 3. 再举例：取指令阶段

假设要做：

```
M[PC] → IR
```

可能拆成几拍：

##### 第 1 拍

```
PC → MAR，Read
```

需要：

```
PCout = 1
MARin = 1
Read = 1
```

微指令：

```
PCout MARin Read MDRin IRin ...
  1     1    1    0    0
```

##### 第 2 拍

```
MDR → IR
```

需要：

```
MDRout = 1
IRin = 1
```

如果微指令里有这两个控制位，就把对应位设成 1。

所以不译法就是：

> **需要哪个控制信号，就把对应位写 1。**

------

#### 4. 不译法的优点

优点很明显：

```
不需要译码，速度快。
```

因为每一位本来就是一个控制信号。

微指令从控制存储器中取出来以后，不用经过复杂译码，直接控制数据通路。

所以早期微程序控制器常用这种方法。

------

#### 5. 不译法的缺点

缺点也很明显：

```
控制信号多，微指令很长。
```

比如 CPU 有 100 个控制信号，那么微操作码字段至少需要 100 位。

但是一条微指令真正用到的控制信号可能只有几个，大量位都是 0。

所以：

```
控制存储器利用率低。
```

PPT 里说：

> 控制信号多，有效的控制信号只有少数，因此控存利用率低。

意思就是，微指令很宽，但每次真正用到的位不多，浪费空间。

------

### 二、字段直接编码法

#### 1. 基本思想

字段直接编码法比不译法更聪明一点。

它不是“一位对应一个控制信号”，而是：

> **把微指令分成若干字段，每个字段经过译码后产生一组控制信号。**

核心原则是：

```
互斥微命令放在同一个字段；
相容微命令放在不同字段。
```

------

### 三、什么是相容和互斥？

#### 1. 相容微命令

相容就是：

> **可以同时进行的操作。**

比如：

```
PCout 和 MARin
```

可以同时有效，因为它们配合完成：

```
PC → MAR
```

再比如：

```
ALUadd 和 RegWrite
```

在某些数据通路中也可能配合使用。

------

#### 2. 互斥微命令

互斥就是：

> **不能同时进行的操作。**

比如 ALU 在同一拍里通常不能既做加法又做减法：

```
ALUadd
ALUsub
ALUor
ALUand
```

这些就是互斥的。

再比如存储器操作：

```
Read
Write
```

一般同一时刻不能既读又写，也属于互斥。

所以这些互斥操作就可以放在同一个字段中，用编码表示选择哪一个。

------

### 四、字段直接编码法举例

假设我们有下面这些控制信号。

#### 一组 ALU 操作

```
无操作
ADD
SUB
OR
AND
```

这些操作互斥，ALU 同一拍只能选一种，所以可以放进一个字段：

| ALU 字段编码 | 含义   |
| ------------ | ------ |
| 000          | 无操作 |
| 001          | ADD    |
| 010          | SUB    |
| 011          | OR     |
| 100          | AND    |

这样原本如果用不译法，可能需要 5 位：

```
ALUadd ALUsub ALUor ALUand ...
```

现在用一个 3 位字段就能表示。

------

#### 一组存储器操作

```
无操作
Read
Write
```

也互斥，可以放一个字段：

| MEM 字段编码 | 含义   |
| ------------ | ------ |
| 00           | 无操作 |
| 01           | Read   |
| 10           | Write  |

------

#### 一组寄存器输入选择

比如：

```
无写入
MARin
IRin
MDRin
RegWrite
```

也可以编码：

| Dest 字段编码 | 含义     |
| ------------- | -------- |
| 000           | 无写入   |
| 001           | MARin    |
| 010           | IRin     |
| 011           | MDRin    |
| 100           | RegWrite |

------

### 五、用字段直接编码法表示一条微指令

假设微指令分成三个字段：

```
ALU字段 | MEM字段 | Dest字段 | 下地址
```

比如要执行：

```
PC → MAR，Read
```

可以编码成：

```
ALU字段 = 000   // ALU 不操作
MEM字段 = 01    // Read
Dest字段 = 001  // MARin
```

但是注意，还需要有一个字段表示谁输出到总线，比如：

```
Source字段 = PCout
```

所以更完整的微指令可能是：

```
Source字段 | Dest字段 | MEM字段 | ALU字段 | 下地址
```

执行：

```
PC → MAR，Read
```

可以写成：

```
Source = PCout
Dest   = MARin
MEM    = Read
ALU    = 无操作
```

这样每个字段经过译码器产生对应控制信号。

## 执行顺序如何控制——增量（计数器）法

下条微指令有四种情况:

- 取指微程序:每条指令一定先执行取指微程序。
- 第一条微指令:取指后转到该指令的第一条微指令。
- 顺序执行时:顺序取下条微指令。
- 分支执行时:根据不同指令选择下条微指令。

微程序控制器需要确定下一条微指令地址，常见方法有计数器法和下址字段法。

计数器法：

- 下址隐含在 μPC 中。顺序执行时 μPC←μPC+1；
- 发生转移时，由起始和转移地址发生器产生新的 μPC。
- 优点是微指令短、节省控制存储器；
- 缺点是转移控制需要额外逻辑，灵活性和速度略差。

下址字段法：

- 微指令中设置明显的“下址字段”，直接指出下一条微指令地址。
- 执行当前微指令时，同时得到下一条微指令地址。
- 优点是速度快、转移方便；
- 缺点是每条微指令都要包含下址字段，占用较多控制存储器空间。

## 异常与中断的处理

- 中断:顺序执行指令流的过程中外部请求而中止执行当前的承程序。
- 异常:顺序执行指令流的过程中内部异常而中止执行当前的程序。

## 异常的分类

![img](C:\Users\22067\Desktop\计组复习\assets\cover423_20260511080003.jpg)异常和中断会打断 CPU 的正常执行流程，使 CPU 转去执行专门的异常处理程序或中断服务程序。

内部异常按处理方式可分为：
1. Fault 故障
   由当前指令执行引起，如缺页、TLB 缺失、溢出等。
   处理后通常返回当前指令重新执行。

2. Trap 自陷
   预先安排的事件，如系统调用、单步跟踪。
   当前指令已经完成，处理后返回下一条指令。

3. Abort 终止
   严重硬件故障，通常无法恢复。
   断点不重要，一般终止当前进程或重启系统。

处理器检测到异常后的步骤：
1. 关中断，防止新中断破坏现场。
2. 保存断点和程序状态，将 PC、PSW 保存到栈或专用寄存器。
3. 识别异常原因，方式包括软件识别和硬件识别。

软件识别：

- 设置异常状态寄存器记录异常原因，如 MIPS 的 Cause 寄存器。
- 异常发生后先进入统一异常处理入口，再由软件查询 Cause 判断具体原因。
- MIPS 采用这种方式。

硬件识别：

- 硬件产生中断类型号，根据类型号查中断向量表，得到对应中断服务程序入口地址。
- 80x86 采用这种方式。

软件识别硬件简单但速度慢；
硬件识别速度快但硬件复杂。

## MIPS带异常和中断处理的数据通路设计

> MIPS 发生异常时，要做 4 件事：**保存断点 EPC、记录异常原因 Cause、关闭中断/修改状态 Status、把 PC 改成异常处理程序入口地址 0x80000180。**

![img](C:\Users\22067\Desktop\计组复习\assets\cover542_20260511080004.jpg)

### MIPS 异常处理中用到的三个寄存器

PPT 说和异常/中断有关的三个寄存器是：

```
EPC
Cause
Status
```

这三个寄存器都属于 MIPS 协处理器 CP0 中的控制寄存器。

------

#### 1.1 EPC：保存断点地址

EPC 全称是：

```
Exception Program Counter
```

意思是：

> **异常发生时，用来保存程序应该返回的位置。**

比如当前正在执行某条指令时发生异常，CPU 要跳去执行异常处理程序。可是处理完以后，它得知道原程序应该从哪里恢复。

这个“回来时的地址”就存在 EPC 里。

例如：

```
EPC ← 发生异常的指令地址
```

或者：

```
EPC ← 下一条指令地址
```

具体存当前还是下一条，要看异常类型。

------

#### 1.2 Cause：记录异常原因

Cause 寄存器用来记录：

```
为什么发生异常？
```

PPT 里圈出来的是：

```
ExcCode
```

也就是 exception code，异常代码字段。

比如你们 PPT 后面假设两种异常：

```
未定义指令：Cause = 0
数据溢出：Cause = 1
```

所以 Cause 的作用就是：

> **告诉操作系统：这次异常到底是哪一种。**

------

#### 1.3 Status：记录处理器状态 / 中断屏蔽位

Status 寄存器记录处理器当前状态。

PPT 圈出来的是：

```
IM[7:0]
```

也就是 Interrupt Mask，中断屏蔽位。

它可以控制哪些中断允许响应，哪些中断被屏蔽。

PPT 写：

```
把 Status 寄存器的中断屏蔽位 IM 赋值为 0，以关闭中断
```

意思是发生异常以后，先把中断关掉，避免新的中断打断当前异常处理过程。

------

#### 2. 第一页下面：硬件要实现哪些功能？

PPT 列了 4 个硬件功能。

------

##### 功能 1：把返回地址写入 EPC

```
EPC ← 返回地址
```

这个返回地址就是断点。

如果异常处理结束后还能回到原程序，就要靠 EPC 找回去。

------

##### 功能 2：把异常原因写入 Cause

```
Cause ← 异常原因
```

比如：

```
未定义指令：Cause = 0
数据溢出：Cause = 1
```

这样异常处理程序就知道该怎么处理。

------

##### 功能 3：关闭中断

```
Status.IM ← 0
```

目的：

> 防止处理异常时，又来一个新中断，把现场搞乱。

------

##### 功能 4：把异常处理程序入口地址写入 PC

MIPS 的异常处理入口地址是：

```
0x80000180
```

所以发生异常后：

```
PC ← 0x80000180
```

这一步非常关键。

因为 CPU 要从正常程序跳到操作系统的异常处理程序。

### MIPS采用软件识别中断源

![img](C:\Users\22067\Desktop\计组复习\assets\cover427_20260511080004-1782536397564-58.jpg)

#### 4.1 EPCWr 信号

PPT 写：

```
EPCWr：在保存断点时该信号有效，使断点 PC 写入 EPC。
```

意思是：

```
EPCWr = 1 时，允许写 EPC
```

类似寄存器组的 RegWr。

如果：

```
EPCWr = 0
```

EPC 不变。

------

#### 4.2 CauseWr 信号

PPT 写：

```
CauseWr：在处理器发现异常时该信号有效，使异常类型被写到 Cause 寄存器。
```

也就是：

```
CauseWr = 1 时，允许写 Cause
```

------

#### 4.3 IntCause 信号

PPT 写：

```
需要一个控制信号 IntCause 来选择正确的值写入 Cause 中
```

因为这套数据通路只考虑两种异常：

```
未定义指令：Cause = 0
溢出：Cause = 1
```

所以 Cause 前面有一个 MUX。

```
IntCause = 0 → 写入 0，表示未定义指令
IntCause = 1 → 写入 1，表示溢出
```

------

#### 5. 为什么写入 EPC 的可能是 PC 或 PC-4？

PPT 第二页有一句很重要：

```
写入 EPC 的断点可能是正在执行的指令地址，也可能是下条指令的地址。
前者需要把 PC 的值减 4 后送到 EPC，后者则直接送 PC 到 EPC。
```

这个地方很容易懵，慢慢拆。

在多周期 MIPS 里，取指阶段通常已经做了：

```
PC ← PC + 4
```

也就是说，等到后面译码、执行阶段发现异常时，PC 里面往往已经不是当前指令地址，而是下一条指令地址了。

比如当前指令地址是：

```
1000
```

取指之后 PC 已经变成：

```
1004
```

如果这条地址 1000 的指令发生异常，并且要保存“当前指令地址”，那就必须：

```
EPC ← PC - 4
```

也就是：

```
EPC ← 1004 - 4 = 1000
```

这就是为什么异常状态里要让 ALU 做减法：

```
PC - 4
```

## 单周期和多周期

核心流程和单周期差不多

但是在起初的响应方面有些许区别

单周期 CPU 的每条指令在一个时钟周期内完成所有操作（取指、解码、执行、访存、写回）。由于整个指令执行过程是原子性的，异常和中断只能在指令执行完毕后才能被检测和响应

多周期 CPU 将指令分解为多个阶段（如取指、解码、执行、访存、写回），每个阶段占用一个时钟周期。这种设计允许在指令执行过程中动态检测和响应异常，而中断仍需等待指令结束：

例如除法指令在执行阶段检测到除数为零时，CPU 可立即终止当前指令的后续阶段（如访存和写回），并跳转至异常处理程序

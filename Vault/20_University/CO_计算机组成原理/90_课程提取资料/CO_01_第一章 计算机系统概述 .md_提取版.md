## 冯诺依曼结构

1. 计算机有运算器、控制器、存储器、输入和输出设备。

2. 各部件的功能:

   - **运算器**:加/减/乘/除四种算术运算和一些逻辑运算;

   - 控制器:执行指令;

   - **存储器**:存放数据和指令,形式相同,控制器区分;

   - 输入和输出设备:人-机通信。

3. 程序由指令序列构成;指令由**操作码和地址码**组成。操作码指出操作类型,地址码指出操作数的地址。指令和数据以二进告制表示。

4. 采用"存储程序"方式。事先编好的程序和数据存入主存存,执行时自动逐条取出指令和执行指令。

## 计算机的基本组成

- 完整的计算机系统由软件和硬件组成。
- 计算机实现的所有任务都是通过执行一条一条指令完成的!
  - 运算器(数据运算):ALU、GPRs、标志寄存器等
  - 存储器(数据存储):存储阵列、地址译码器、读写控制电路
  - 总线(数据传送):数据(MDR)、地址(MAR)和控制线
  - 控制器(控制):对指令译码、生成控制信号
  - I/O设备

## 计算机的基本部件与指令执行流程

![image-20260622211610995](C:\Users\22067\AppData\Roaming\Typora\typora-user-images\image-20260622211610995.png)

### 1. PC：程序计数器

PC 里面放的是**下一条要执行的指令地址**。

比如：

```
PC = 0000H
```

意思是：CPU 接下来要去内存的 `0000H` 地址取指令。

------

### 2. MAR：存储器地址寄存器

MAR 用来暂存**要访问的内存地址**。

无论是去内存取指令，还是去内存取数据，都要先把地址放进 MAR。

比如：

```
MAR = PC = 0000H
```

然后 MAR 通过地址总线告诉内存：

```
我要访问 0000H 这个地址
```

------

### 3. MDR：存储器数据寄存器

MDR 用来暂存**内存和 CPU 之间传输的数据**。

如果是读内存：

```
内存 → 数据总线 → MDR
```

如果是写内存：

```
MDR → 数据总线 → 内存
```

所以 MDR 是 CPU 和内存交换数据时的“中转站”。

------

### 4. IR：指令寄存器

IR 用来保存**当前正在执行的指令**。

比如内存中取出来的指令是：

```
ADD R1, R2, R3
```

这条指令会先进入 MDR，再送入 IR。

IR 保存这条指令后，控制器才能分析它。

------

### 5. GPRs：通用寄存器组

GPRs 里面放的是 CPU 常用的数据，比如：

```
R1 = 5
R2 = 8
R3 = 0
```

ALU 做运算时，通常直接从 GPRs 里取数。

------

### 6. ALU：算术逻辑单元

ALU 负责真正计算，比如加法、减法、与、或、比较等。

例如：

```
R1 + R2 = 5 + 8 = 13
```

这个加法就是 ALU 做的。

现代计算机完成取指令和执行指令的步骤是这样的：

- **取指令：**将PC中存放的指令地址放入MAR中，再通过MAR和地址总线去内存中寻找该地址对应的指令。找到后将指令数据通过数据总线从内存传入MDR，同时PC指向下一条指令。

- **执行指令：**MDR将指令数据（如ADD R1，R2）传入IR，IR会拆分指令的操作码和操作数地址，将其传入控制器，控制器会将生成的控制信号传给ALU以及控制总线。若GPRs中存放了所需操作数则直接将操作数通过ALU进行计算得到结果存回GPRs或者按需写入内存（通过MDR中转），并更新标志寄存器。若GPRs中没有存放所需操作数，则还需先通过MAR去内存中寻找操作数地址对应的操作数在通过数据总线将其通过MDR中转传如GPRS中

假设 PC 里面现在是：

```
PC = 0000H
```

内存中：

```
0000H: ADD R3, R1, R2
0004H: SUB R4, R3, R1
```

那么取指令过程是：

### 第一步：PC 把地址送给 MAR

```
MAR ← PC
```

也就是：

```
MAR ← 0000H
```

意思是：CPU 准备访问内存地址 `0000H`。

------

### 第二步：MAR 通过地址总线访问内存

```
地址总线 = 0000H
```

内存看到地址后，就找到 `0000H` 里面存的内容。

这个内容是：

```
ADD R3, R1, R2
```

------

### 第三步：内存把指令送入 MDR

```
MDR ← Memory[MAR]
```

也就是：

```
MDR ← ADD R3, R1, R2
```

这一步是通过数据总线完成的。

------

### 第四步：MDR 把指令送入 IR

```
IR ← MDR
```

也就是：

```
IR ← ADD R3, R1, R2
```

现在 IR 里面保存的就是当前要执行的指令。

------

### 第五步：PC 指向下一条指令

如果一条指令占 4 字节，那么：

```
PC ← PC + 4
```

原来：

```
PC = 0000H
```

现在变成：

```
PC = 0004H
```

这样下一次取指令时，CPU 就会去取下一条指令。

------

## 三、执行指令过程：分析并完成这条命令

现在 IR 里面有：

```
ADD R3, R1, R2
```

意思是：

```
R3 ← R1 + R2
```

也就是把 R1 和 R2 的值相加，结果放到 R3。

假设：

```
R1 = 5
R2 = 8
```

执行过程如下：

------

### 第一步：IR 把指令交给控制器分析

控制器会看这条指令的操作码和寄存器编号。

它会分析出：

```
操作类型：ADD，加法
源操作数1：R1
源操作数2：R2
目的寄存器：R3
```

也就是说，控制器知道了这条指令要干什么。

------

### 第二步：从 GPRs 中读出 R1 和 R2

```
A ← R1 = 5
B ← R2 = 8
```

这两个数会被送到 ALU 的两个输入端。

------

### 第三步：控制器让 ALU 执行加法

控制器给 ALU 一个控制信号：

```
执行 ADD 运算
```

于是 ALU 计算：

```
5 + 8 = 13
```

------

### 第四步：结果写回 GPRs

```
R3 ← 13
```

于是通用寄存器组变成：

```
R1 = 5
R2 = 8
R3 = 13
```

这条 ADD 指令就执行完了。

## 编程工具层次化与 ISA 

### 1. 程序执行的基本过程

高级语言程序不能被硬件直接执行，最终都要转换成**机器指令**。

```
高级语言程序 → 编译程序 → 汇编语言程序 → 汇编程序 → 机器语言程序 → CPU执行
```

核心结论：

**任何高级语言程序，最终都是通过执行若干条机器指令完成的。**

------

### 2. 三种编程层次

| 编程方式     | 特点                                   |
| ------------ | -------------------------------------- |
| 机器语言编程 | 直接使用 0 和 1 编写，硬件可直接识别   |
| 汇编语言编程 | 用助记符表示机器指令，需要汇编程序转换 |
| 高级语言编程 | 接近人的表达，需要编译器或解释器转换   |

------

### 3. 编译程序与汇编程序

**编译程序：**

```
高级语言源程序 → 汇编语言程序 / 机器语言程序
```

**汇编程序：**

```
汇编语言程序 → 机器语言程序
```

------

### 4. 机器指令的组成

每条机器指令通常由两部分组成：

```
操作码 + 操作数 / 地址码
```

其中：

```
操作码：说明做什么操作
操作数 / 地址码：说明对谁操作，数据在哪里
```

例如：

```
ADD R1, R2, R3
```

表示：

```
R1 ← R2 + R3
```

------

### 5. ISA 的定义

ISA 全称：

```
Instruction Set Architecture
指令集体系结构
```

ISA 是**软件和硬件之间的接口**，规定了软件如何使用硬件。

简单理解：

```
ISA 是一套规则，规定 CPU 能执行哪些指令，以及指令怎么写、怎么执行。
```

------

### 6. ISA 规定的主要内容

ISA 主要规定：

```
1. 可执行的指令集合
2. 指令格式
3. 操作类型
4. 操作数类型
5. 寄存器的名称、编号、长度和用途
6. 存储空间大小和编址方式
7. 大端 / 小端存储方式
8. 寻址方式
9. 指令执行流程和控制方式，如 PC 的使用
```

------

### 7. ISA 与软件、硬件的关系

```
软件建立在 ISA 之上
硬件负责实现 ISA
```

ISA 位于软件和硬件的交界面，是对硬件的抽象。

也就是说：

```
软件只需要按照 ISA 写指令；
硬件只需要保证能正确执行这些指令。
```

------

### 8. ISA 和计算机组成的区别

| 项目       | 关注点                     |
| ---------- | -------------------------- |
| ISA        | 规定 CPU 应该提供什么功能  |
| 计算机组成 | 研究这些功能如何用硬件实现 |

例如：

```
ISA 规定要有加法指令；
计算机组成研究如何用 ALU 实现加法。
```

------

### 9. ISA 与计算机组成的关系

重要结论：

```
1. 不同计算机可以有不同 ISA，如 x86、MIPS、ARM、RISC-V。
2. 计算机组成必须实现 ISA 规定的功能。
3. 同一种 ISA 可以有不同的硬件实现。
```

例如，同样是 RISC-V ISA，可以设计成：

```
单周期 CPU
多周期 CPU
流水线 CPU
```

------

### 10. 用户层次化

不同用户面对的层次不同：

```
最终用户：使用应用程序
系统管理员：管理操作系统
应用程序员：使用高级语言和编译器
系统程序员：需要理解 ISA、汇编语言和底层硬件
```

## 系统性能评价（必考）

### 计算机性能的测量

**吞吐率：**单位时间内系统完成的工作量或处理的任务数量

**响应时间：**从用户或系统发起请求开始，到接收到完整响应所耗费的时间 ，涵盖请求传输、系统处理、结果返回的全过程

**CPU的执行时间：**执行程序中每条指令的时间

### CPU执行时间的计算——时钟和CPI

CPI=总时钟周期数/指令条数（每一条指令的**时钟周期数N**）

> - 不同的指令需要的周期数(CPI)可能是不同的。
> - 对于某一指令而言,其CPI是一个确定的值。
> - 对于某一程序或机器而言,其CPI是一个平均值,表示该程序或该机器每条指令平均需要多少时钟周期。

CPU执行时间=CPU总时间周期数 x 时钟周期 T

​                       =CPU总时间周期数 / 时钟频率

​                       =CPI x 时钟周期T x 指令数

### 如何计算CPI和CPU时间

#### 1. 按指令类型计算总时钟数

假定程序中第 $i$ 类指令的 CPI 为 $CPI_i$，指令条数为 $C_i$，则：

$$
总时钟数 = \sum_{i=1}^{n}(CPI_i \times C_i)
$$

因此：

$$
CPU时间 = 时钟周期 \times \sum_{i=1}^{n}(CPI_i \times C_i)
$$

---

#### 2. 计算程序综合 CPI

假定 $CPI_i$ 为第 $i$ 类指令的 CPI，$F_i$ 为第 $i$ 类指令在程序中的出现频率，则程序综合 CPI 为：

$$
CPI = \sum_{i=1}^{n}(CPI_i \times F_i)
$$

其中：

$$
F_i = \frac{C_i}{Instruction\_Count}
$$

说明：

- $CPI_i$：第 $i$ 类指令的 CPI
- $C_i$：第 $i$ 类指令的条数
- $F_i$：第 $i$ 类指令的出现频率
- $Instruction\_Count$：程序总指令条数

---

#### 3. 根据 CPU 时间反推 CPI

已知 CPU 时间、时钟频率、总时钟数、指令条数，则程序综合 CPI 为：

$$
CPI = \frac{CPU时间 \times 时钟频率}{指令条数}
$$

也可以写成：

$$
CPI = \frac{总时钟周期数}{指令条数}
$$

---

#### 4. 核心公式总结

$$
CPU时间 = 指令条数 \times CPI \times 时钟周期
$$

$$
CPU时间 = \frac{指令条数 \times CPI}{时钟频率}
$$

$$
CPI = \frac{总时钟周期数}{指令条数}
$$

$$
总时钟周期数 = 指令条数 \times CPI
$$

#### MIPS（每秒执行几百万条指令数）

MIPS 表示每秒执行多少百万条指令：
$$
MIPS=\frac{C}{t \times 10^6}
$$
其中：

\- $C$：指令条数
\- $t$：执行时间
\- $CPI$：平均每条指令所需时钟周期数
\- $f$：时钟频率

CPU 执行时间为：
$$
t=C \times CPI \times \frac{1}{f}
$$
即：
$$
t=\frac{C \times CPI}{f}
$$
代入 MIPS 公式：
$$
MIPS=\frac{C}{\frac{C \times CPI}{f} \times 10^6}
$$
化简得：
$$
MIPS=\frac{f}{CPI \times 10^6}
$$
如果 $f$ 的单位是 MHz，则：
$$
MIPS=\frac{f}{CPI}
$$

> 用MIPS评价计算机性能不可靠。
>
> 因为不同的指令的执行代价悬殊。
>
> MIPS 仅统计指令数量，却忽略了指令的「执行复杂度」。例如：
>
> - 简单指令（如加法`ADD`）可能仅需 1 个时钟周期；
> - 复杂指令（如内存访问`LOAD`、乘法`MUL`）可能需要多个周期（甚至跨时钟域等待）。 案例：假设 CPU A 执行 100 条简单指令耗时 1 秒（100 MIPS），CPU B 执行 50 条复杂指令也耗时 1 秒（50 MIPS），但后者的实际计算量可能远超前者。
>
> 所以相同时间内执行指令数少（MIPS小）的CPU计算的计算量不一定少，性能也不一定差

#### 综合评价结果（结合运行时间）

先看一个例子:

- 程序1:在机器A上执行需要1秒,在B上需要10秒
- 程序2:在机器A上执行需要1000秒,在B上需要100秒

你的结论是什么?

1. 比较机器性能，最可靠的是比较总执行时间。
2. 执行时间越短，性能越好。
3. 多个程序综合评价时，要考虑每个程序的使用频率，即加权平均。
4. 算术平均适合计算平均执行时间。
5. 几何平均适合比较相对性能、加速比、规格化执行时间。
6. 几何平均不能直接得到程序总执行时间。
